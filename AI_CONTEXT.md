# Underwriting Copilot — Context Brief for an AI Model

This document explains the web app in `output/underwriting_copilot_mvp.html` (generated by `src/dashboard.py`) so another AI can reason about it without reading all the code.

## 1. What it is

Underwriting Copilot is an **AI-assisted life-insurance underwriting workbench**. It takes life-insurance application "packets," screens them for internal contradictions, scores each applicant's risk, and issues an Approve / Manual-Review / Decline decision with a written rationale. A human underwriter makes the final call on the borderline cases; clear-cut cases are decided automatically. It is a **private prototype built on synthetic data** — not a production system and not connected to any real insurer.

## 2. Two layers: a Python pipeline and a single-file web app

**Backend pipeline (Python, `src/`)** runs offline and produces the data the app displays:
- `datagen.py` — generates synthetic applicants with correlated risk factors and a ground-truth "high risk" label.
- `external_data.py` — trains small models on 17 public health/credit datasets and blends them into an "External Risk Prior" fed into the main models.
- `published_models.py` — Framingham office-based CVD model (published coefficients, no training) exposed as the `published_cvd_prior` feature.
- `docgen.py` — builds 5-document PDF packets (application form, payslip, paramedical report, 3-month bank statement, tax slip) and **injects contradictions at a 30% rate** so detection can be measured.
- `extract.py` — pulls key fields out of the packets.
- `engine.py` — the core: conflict detection (6 checks), a weighted **rule engine**, **ML models** (logistic regression + gradient boosting), the **affordability screen** (4 financial-viability indicators + indicative premium estimator), and the **decision logic**.
- `run_pipeline.py` — orchestrates the above and writes `output/portfolio.json` (all cases) + `output/evaluation_report.json` (metrics).
- `dashboard.py` — injects `portfolio.json` into a big HTML/CSS/JS template and writes the final self-contained `output/underwriting_copilot_mvp.html`.

**Frontend (the HTML file)** is a **self-contained, client-side single-page app** written in vanilla JS. All case data is embedded at build time as a JS constant (`const DATA = {...}`). There is no server, no network calls, no framework. It runs entirely in the browser. User actions persist to `localStorage`.

## 3. The data model (one "case")

Each case object (from `portfolio.json`, embedded as `DATA.cases`) has roughly these fields:

- Identity/application: `id` (e.g. "APP-1123"), `name`, `age`, `occupation`, `city`, `state`, `policy`, `coverage` (face amount, USD), `has_docs`.
- Risk inputs: `bmi`, `smoker`, `conditions`, `credit`, `dti` (debt-to-income), `ext_prior`, `pub_prior` (Framingham CVD).
- Affordability: `premium` (indicative annual premium) and `afford` — `{verdict: "pass"|"strain"|"fail", label: "AFFORDABLE"|"STRAINED"|"NOT JUSTIFIED", premium, pti, disposable, cov_mult, cov_cap, dsr, indicators[], reasons[]}`. A `fail` verdict forces MANUAL REVIEW ("Referred — Financial Underwriting Review") regardless of risk score.
- Scores: `risk_score` (composite 0–100), `rule_score`, `ml_score` (gradient boosting), `ml_score_lr` (logistic), `rule_factors` (array of `[label, detail, points]`).
- Screening: `conflicts` (array of `{type, severity: "major"|"minor", description}`), `unique` (disclosed circumstance text or null).
- Decision: `verdict` ("green"|"yellow"|"red"), `decision` ("APPROVE"|"MANUAL REVIEW"|"DECLINE"), `rate_class`, `reasons` (array of strings), `label` (ground truth), `ai_summary` (pre-written narrative).

## 4. Scoring and decision logic

**Composite risk score** = `0.5 × rule_engine + 0.5 × ML_probability`, on a 0–100 scale.

**Bands (thresholds are STP-optimised per pipeline run, fallback 50 / 90):**
`run_pipeline` grid-searches the approve/decline lines to maximise straight-through
processing under safety constraints (approve-zone true-risk ceiling, decline-zone
precision floor) and writes them to `metrics.decisioning.thresholds`; the app reads
`A_LINE`/`D_LINE` from there (this build: 34 / 60).
- `< A_LINE` → **APPROVE** (green) — clear-cut, auto-approved.
- `A_LINE–D_LINE-1` → **MANUAL REVIEW** (yellow) — a human underwriter must look at it.
- `≥ D_LINE` → **DECLINE** (red) — risk exceeds appetite.

**Overrides that force manual review or decline regardless of score:**
- Any **major cross-document conflict** → at least manual review.
- **Material misrepresentation** (smoker non-disclosure or DOB mismatch) → automatic **DECLINE**.
- **Model disagreement** (rule vs ML differ by >20 points) → manual review.
- A **disclosed unique circumstance** → manual review.
- A **failed affordability screen** (NOT JUSTIFIED) → manual review, "Referred — Financial Underwriting Review".

Important implementation detail: the app **recomputes every case's verdict/decision/reasons client-side at load** (`recomputeVerdicts()` in the JS, using the pipeline's exported thresholds and the baked `afford` verdict) to stay consistent — it does not rely solely on the values baked into `portfolio.json`.

## 5. Two roles (demo credential login)

Login is a simple front-end credential check against a hard-coded `ACCOUNTS` map (demo only, no real auth). Four accounts:

- **Senior underwriter** — `mrivera` / `senior`
- **Mid-tier underwriter** — `ewong` / `review`
- **New analyst** — `dpark` / `analyst`
- **Manager** — `rkubusada` / `oversight`

**Underwriters** work cases. **Managers** only oversee — their sidebar shows just "Portfolio & Model Card" and "Manager Overview." Managers have no case queue, no records, no ability to action cases.

## 6. Case spaces (the underwriter's workspace)

Cases are bucketed by `bucketOf(case)`:
- **Review Queue** — yellow/manual-review cases not yet decided. **These are the only cases underwriters act on.**
- **Completed** — manual-review cases the underwriter has approved/declined.
- **Auto-Approved** — green cases the system approved automatically (read-only record).
- **Auto-Declined** — red cases the system declined automatically (read-only record).

## 7. The review workflow (per case)

The Review Queue is **ranked by a priority score** (`priorityScore`) combining coverage/exposure, risk severity, number/severity of conflicts, disclosed circumstances, and how long the case has aged. Bands: CRITICAL / HIGH / MEDIUM / LOW.

**Tiered auto-assignment** (`assignTier`): each case is routed to an underwriter by complexity — high-exposure/conflict/critical → the **senior**, medium → the **mid-tier**, simplest → the **new analyst**. By default an underwriter sees only cases assigned to them ("My cases"), with a toggle to see the "Whole team."

**SLA timer:** each case has a seeded `receivedAt` timestamp. The app shows time-in-queue live (refreshes each minute). Under 6h neutral, 6–8h amber, **over 8h flips red as an SLA breach.**

**Actions an underwriter can take** on a review-queue case (each writes to an audit trail with name, role, timestamp):
- Claim / take over (assignment)
- **Approve** or **Decline** (both require a typed rationale)
- **Request info** (NIGO — sends it back as info-requested)
- Add a note
- After deciding, the case leaves the queue and moves to **Completed**; the decision also feeds an "overrides" store used for model retraining. It can be Reopened.

## 8. State & persistence

All user actions live in the browser via `localStorage`:
- `uw_workbench` — per case: `{status, assignee, assigneeUid, tier, receivedAt, notes[], history[], decision, pulled}`.
- `uw_overrides` — human decisions, exportable to `data/overrides.json` so the Python pipeline can retrain on them.

Statuses: `new → in_review → info_requested → referred → approved → declined`.

## 9. What a case page shows

Tabs per case: Application, Documents, Extraction & Conflicts, Risk Score (composite gauge + rule-factor breakdown + the 0·50·90·100 band bar), and Decision (system verdict with bulleted "why," the underwriter Case Desk with actions, the audit trail, and a grounded AI summary). There is also a "Score New Application" tool that reads an uploaded PDF, extracts fields, and scores it live in the browser using the exported logistic-regression coefficients.

## 10. Design / tech notes

- UI style is modeled on the "MediFlow" healthcare dashboard: **dark theme, indigo (#6A67F7) accent, Poppins font, large rounded cards, pill-shaped buttons and chips.**
- No build step for the frontend beyond running `python src/dashboard.py`. No external JS libraries except `pdf.js` (loaded from CDN) for reading uploaded PDFs.
- Caveats for reasoning: data is **synthetic** (so model accuracy is an upper bound), verdicts are **recomputed client-side**, and all workflow state is **local to the browser** (clearing storage resets assignments, timers, and decisions).
