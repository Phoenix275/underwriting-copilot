# PRD — Underwriting Copilot v2

**Status:** Draft for review · **Date:** 2026-07-23
**Sources:** Team meeting notes (July 2026), stakeholder change requests, product deep-research (cited in §8)
**Owner:** Tegh Bindra · Tech Mahindra GLP — Finance Project 1

## 1. Background

The Underwriting Copilot reads a life-insurance application packet, screens it for
cross-document contradictions, scores mortality risk with a dual engine (auditable
rules + ML), screens financial viability, and routes what it cannot decide to the
right underwriter desk. It is deployed as a single self-contained HTML dashboard at
**underwriting-copilot.pages.dev**, generated by `src/dashboard.py` from the Python
pipeline's output. A richer React workbench exists in `web/` (kept in CI, not
deployed).

This PRD consolidates every change agreed in the July team meeting plus subsequent
stakeholder requests into one prioritized, implementable specification.

**The product's one-line story** (meeting ask: "craft a good storyline"): *a
transparent underwriting copilot that automates the clear cases, explains every
point of its score with published evidence, and routes the hard cases to the right
human — with the audit trail regulators ask for.*

## 2. Goals / non-goals

**Goals**

1. Make the review queue read like an underwriter's worklist: ranked by what
   matters (coverage, time in queue, risk), with the decision and risk score
   visible while scrolling.
2. Make every screen navigable and auditable: back buttons, case IDs first-class,
   reviewer recommendations visible on reviewed cases.
3. Give management its own instruments: an executive view of money and mix, an
   admin view of the decision/audit trail, bulk actions with guardrails.
4. Encode real underwriting-requirements logic (the APS age × amount grid) so the
   copilot states *what evidence a case requires*, not just a score.
5. Recommend the right underwriter for each referral from availability, workload,
   and authorization level.
6. Keep the model defensible: every weight evidence-anchored, the rule layer
   documented, and a clear adaptation story for regime shocks (COVID-class events).

**Non-goals (this cycle)**

- Real authentication or production PII handling (demo remains synthetic).
- Live calendar integration (Google/Outlook) — v2 ships a workload model with
  demo availability data; the integration contract is specified for v3.
- Extending to non-life products (tracked as an open question, §10).
- Retina/ocular biometrics as an underwriting input (research verdict in §8; it
  does not survive contact with the evidence as a near-term feature).

## 5. Detailed requirements

### Epic A — Queue & navigation

**A1. Ranked review queue (P0).**
The review queue is ordered by a transparent priority score. Keep the existing
blend — face amount, risk score, conflict severity, disclosed circumstances, hours
in queue — but:
- Show the three headline factors on each row: **coverage**, **time in queue**
  (or SLA state), **risk score**. These are the ranking inputs the team named.
- A column-header sort control per factor (rank stays default).
- The priority formula is documented in "How this score works" so an underwriter
  can answer *why is this case above that one*.

Acceptance: opening the review queue shows rank, ID, coverage, time-in-queue,
risk, decision chip; clicking a column re-sorts; default order restores.

**A2. Case ID first-class; new ID column (P0).**
- Queue rows: the case ID (`APP-1125`) becomes the row's primary identifier
  (larger, monospace); applicant name drops to secondary text.
- Review/manager tables gain a dedicated, sortable **ID** column.
- Case file header: ID rendered above the name at display size; the name remains
  but visually subordinate (meeting: "Case ID should be bigger / Name should be
  smaller").

**A3. Back button on every page (P0).**
Every non-root view renders a `← Back` control in a consistent position
(top-left, above the page title):
- Case file → returns to the queue/space the user came from (preserving scroll
  position and active filters).
- Executive / admin / manager / overview → returns to the previous view.
- Implementation: an in-app navigation stack (the dashboard is a single file;
  browser history integration is a nice-to-have, the in-app stack is the
  requirement).

**A4. Remove "Score New Application" (P0).**
Delete the view, its nav entry, and its dead code. Rationale: it distracts from
the core review workflow, duplicates pipeline scoring in a second code path that
can drift, and no persona in §3 needs it. (The React app retains its equivalent
for the API demo; that app is not deployed.)

### Epic B — Case file

**B1. Risk score and decision at the top (P0).**
The case header leads with: case ID, the composite risk score (large, with band
color), the system decision (APPROVE / MANUAL REVIEW / DECLINE), and the
affordability verdict. Meeting note verbatim: "Risk score should be at top — not
just manual review / accept / decline."

**B2. Top-3 risks to the person (P0).**
Directly under the header: the three highest-weighted rule factors for this case
(from `rule_factors`, sorted by points, ties broken by factor severity), each with
its point contribution and one-line explanation. This is the 10-second read an
underwriter starts from.

**B3. Request-more-docs, in the documents panel, with rationale (P1).**
Move the existing NIGO/request-info control into the "Read from the packet"
section — the place an underwriter notices something missing. Requesting docs
requires: which document(s), and a rationale (min 5 chars, same rule as decisions).
The request is written to the case history **and** surfaces in the admin feed (C2).

**B4. Reviewer recommendation trail (P0).**
For any case with recorded workflow history (claimed, overridden, decided,
info-requested):
- The queue row shows a **recommendation chip** — the most recent reviewer action
  (e.g. ✓ Approved · mrivera).
- The case file shows the full trail (who, what, when, rationale) near the top,
  before the model's own reasoning — the human record outranks the model's.

### Epic C — Management surfaces

**C1. Executive dashboard (P1).**
A view for the executive persona:
- **Total coverage accepted** (approved face amount) and **total declined**.
- **Approval / decline / referral mix** as percentages of the book.
- **Appetite tracking**: approved coverage vs. a configurable monthly target
  ("% compared to how much monthly coverage wanted to take on").
- **Month-over-month**: total risk underwritten in month *N* vs *N+1* (meeting:
  "month of 25 vs month of 26").
- YOY approval-rate line (synthetic seed until multi-year data exists).
The React app already implements most of this; port its metric definitions into
the deployed dashboard.

**C2. Admin feed (P1).**
One screen listing, newest first, across all cases: recorded decisions
(action, rationale, decided_by, timestamp) and **info/doc requests** (B3).
Meeting note: "Admin interface if underwriter requests more info." Each entry
links to its case.

**C3. Bulk approve/decline (P2).**
Multi-select in the review queue with a bulk action bar. Guardrails, which are the
actual spec here:
- Only cases whose system decision matches the bulk action are selectable by
  default (bulk-approving a red case requires an explicit per-case override).
- A single shared rationale is required and is stamped onto every case's trail.
- The action is recorded per-case in the audit trail as `BULK_*` so it is
  distinguishable from individual review.
- Hard cap per action (e.g. 25) to keep the trail meaningful.

### Epic D — Underwriter availability & assignment (meeting §1)

**D1. Workload & availability model (P1).**
Each underwriter gets: current open-case count, capacity (configurable), an
availability state (available / busy / out), an authorization level (max face
amount they may sign), and specializations (e.g. medical-complex, financial).
V2 seeds this from demo data; the data contract is designed so a real
calendar/workload system can populate it later (v3: the meeting's
"integrate with the underwriting team's calendar" ask).

**D2. Assignment recommendation (P1).**
For each referred case, recommend the best underwriter by scoring: authorization
level covers the face amount (hard filter) → specialization match → lowest
workload ratio → availability. Display as "Recommended: E. Wong (review desk,
3/8 open, authorized to $500k)" with the reasons, and let a manager reassign in
one click. The current difficulty→desk routing stays as the first stage; D2
picks the person within the desk.

### Epic E — APS requirement logic (meeting §2)

**E1. Requirements engine (P1).**
Encode the meeting's age × face-amount grid as data (not code), evaluated per
case:

| Age band | $100k–250k | $250k–500k | $500k–999k | $1M+ |
|---|---|---|---|---|
| 0–17 | — | Pediatric exam | Pediatric exam | Pediatric exam |
| 18–40 | — | ExamOne · LabOne · blood profile | ExamOne · LabOne · blood profile | ExamOne · LabOne · blood profile · **APS** · EKG |
| 41–50 | Parametric exam | Parametric exam · blood profile · EKG | Parametric exam · blood profile · EKG · full exam | Parametric exam · blood profile · EKG · full exam |
| 51–60 | Full exam · blood profile · ExamOne | **APS** · full exam · blood profile · ExamOne | **APS** · full exam · blood profile · ExamOne | **APS** · full exam · blood profile · ExamOne · EKG |
| 61–70+ | Full physical · blood profile · ExamOne · EKG · **APS** | Full physical · blood profile · ExamOne · EKG · **APS** | Full physical · blood profile · ExamOne · EKG · **APS** · cognitive assessment | Full physical · blood profile · ExamOne · EKG · **APS** · cognitive assessment |

- The case file shows a **requirements checklist**: what this applicant's age and
  requested amount trigger, with each item's status (on file / outstanding).
- An outstanding APS integrates with B3 (request-more-docs) and links the
  standard form (DOL CA-20).
- The grid lives in one editable table (`src/aps_requirements.py`) so
  underwriting leadership can tune bands without code changes.

**E2. APS-avoidance signal (P2, research-informed).**
Where the grid demands an APS, show what accelerated-underwriting evidence could
substitute (Rx history, MIB, lab data) per the research in §8 — as an
informational panel, not an automated waiver. The meeting's retina-scan idea is
addressed in §8/§10.

### Epic F — Model defensibility & the storyline (meeting §3)

Most of §3's asks are already true of the system; v2's job is to make them
*visible in the product*:

- **"Clear decision-making rule guideline"** → "How this score works" gains the
  full decision policy: composite = 50% rule engine + 50% gradient boosting
  (confirmed in `engine.py` — the meeting asked "check if true"; it is), the
  approve/decline thresholds and how they were tuned on a held-out half, and the
  override rules (conflicts, model disagreement > 20 pts, disclosures force
  manual review regardless of score).
- **"Risk weightage meticulously explained and evidence-based"** → surface the
  provenance table: every medical weight is `round(28 × ln(relative mortality))`
  from published cohorts (NHANES+NCHS linked mortality fit on 20,435 adults /
  2,293 deaths; ERFC; PSC; NHIS), cross-validated on real Prudential applicants
  (AUC ≈ 0.68–0.72). Published as a versioned HF dataset. Answering the meeting's
  "who wrote the dataset weights — the model or team": neither hand-picked —
  derived from evidence, committed as code.
- **"Why gradient boosting"** → one panel: monotone, tabular-data SOTA,
  calibrated probabilities; and why the browser port uses logistic regression
  (coefficients ship, ensembles don't).
- **"Rule machine that sits on top of the model"** → this is the architecture
  today (conflict screen + rule engine + threshold guardrails wrap the ML); name
  it in the product and in the storyline.
- **"Adapt to unforeseen scenarios (COVID) without full retrain"** → documented
  adaptation ladder: (1) rule/threshold layer moves first (a day), (2) score
  recalibration on recent labels (a week), (3) full retrain last (the pipeline
  retrains in ~7 s on this scale, but the *policy* ladder is what a client
  buys). Research on how carriers actually handled COVID in §8.
- **"Hours that go into underwriting"** → add cycle-time capture: time from
  referral to recorded decision, shown per case and aggregated on the executive
  view.
- **Fraud** → the conflict screen *is* an anti-misrepresentation screen
  (income, tobacco/cotinine, DOB, debt, deposits, tax). v2 frames it that way
  and adds the fraud-rate context from research (§8). Deeper claims-fraud
  scoring is out of scope (non-goal) but the hooks are named.

## 6. Prioritization & phasing

**Phase 1 (P0 — the demo-visible UX pass):** A1–A4, B1, B2, B4.
One coherent release: queue reads right, case file leads with the decision
evidence, navigation works, dead feature removed.

**Phase 2 (P1 — the management + requirements release):** C1, C2, B3, D1, D2, E1.
Executive and admin get their surfaces; referrals get named-person assignment;
the APS grid ships.

**Phase 3 (P2 — the polish + research-informed layer):** C3 (bulk), E2
(APS-avoidance panel), cycle-time analytics, storyline content pass (F).

Every phase keeps the invariants: single-file artifact, no external origins at
runtime, evidence-anchored weights untouched, demo logins fictional.

## 7. Success metrics

- **Time-to-first-signal:** an underwriter opening a case sees ID, risk score,
  decision, and top-3 risks without scrolling (measured at 1280×800).
- **Queue scanability:** decision + risk + coverage + queue-time visible for
  every row; zero clicks to read them.
- **Navigation:** every page reachable and leavable without the browser back
  button; no dead ends.
- **Audit completeness:** 100% of decisions, overrides, bulk actions, and doc
  requests carry actor + timestamp + rationale and appear in the admin feed.
- **Assignment quality (demo):** every referral shows a recommended underwriter
  whose authorization covers the face amount.

## 9. Rollout & implementation notes

- All Phase-1 items are changes to `src/dashboard.py` (the deployed artifact) —
  no pipeline changes required. B2 consumes `rule_factors` already present in
  `portfolio.json`.
- E1 adds `src/aps_requirements.py` + pipeline output field `requirements` per
  case; the dashboard renders it. Tests: grid boundary cases (age 40/41, 50/51,
  60/61; $250k/$250,001; $999,999/$1M).
- D1/D2 add an `underwriters.json` demo dataset (availability, capacity,
  authorization, specialization) and an assignment function in the pipeline with
  unit tests; the dashboard displays the recommendation.
- C1/C2 port metric definitions from the React app (`ExecutiveDashboard.tsx`,
  `DecisionsAudit.tsx`) into the dashboard's template.
- The React app in `web/` stays in repo/CI; any feature ported to the deployed
  dashboard should cite the React implementation as reference, not duplicate its
  bugs.
